Overview

  • The camera system receives data from the back cameras as a "Superframe" which is an assembly of the wide angle lens image, depth image, color sensor image (referred to as the 4 MP sensor) and metadata in a single, large YUV420SP format frame, as shown below (scaled down for display).
  • Superframes are in YUV420 SP format, with the UV-plane having VUVU.. ordering 
                                
    • Brief Explanation of YUV420 SP
    • The UV plane is subsampled, so V1 and U1 are the samples corresponding to Y1, Y2, Y7 and Y8.
  • Below shows the format of the Y-plane of the data

    • NB: The ENTIRE Superframe is one YUV420SP image.  That means all the Y-plane data (i.e. all data areas) are first, then the corresponding UV plane areas follow (UV for the full superframe because it is one YUV420SP image). 
    • For Firmware Version 2013.12.11;SF1.0 and later, the ordering in the UV plane is VUVU...  (See this page on how to check version by using the Test Myriad Service App)
    • Just as the color camera's Y-channel data is at the end of the Y-plane, the corresponding UV data for the 4 MP will be at the end of the UV plane.
  • Attached at the bottom of the page is a detailed specification
  • The C struct code under the diagram is the "ground truth" for positioning
  • For functional C++ code see the github repositories
  • All timestamps in the superframe occur in the same clock domain and are highly accurate relative to each other.
  • The depth and large RGB image do co-occur in a single frame, but they will have different timestamps for when they were actually captured by the hardware.  These are the most accurate, and can be used to relate them to the poses of the CVcam image.   Basically we don't attempt to synchronize capture, but do attempt to provide an accurate timeline via the timestamping.
  • We don't recommend parsing superframe data.  Instead use the APIs provided or data log files.  The format is not guaranteed to remain the same, and semantics of variables may change, and variables may be added.  Generally the layout and existing data positions will remain.
  • We really don't recommend parsing superframe data.  However, if it's necessary see the bottom of this page for sf2-parser.tgz, which is a standalone example of how to parse the data.  YUV/Color parsing is a TODO.
  • When accessing superframes from Java, using setPreviewFormat() with NV21 will give the superframe as formatted in this document.  When setting setPreviewFormat() YV12, the UV will no longer be interleaved, but split into separate U & V planes.


Data Segment Ordering & Details


The superframe can be thought of as a "vertical" stack of data. The ordering of the data segments can be seen in the Y-plane area of the superframe (see below for a few examples). Superframe data consists of:
  1. Meta Data areas (IMU (Inertial Measurement Unit), timing, and feature tracking data)
    • IMU data is taken at different data rates:
      • Accelerometer data samples occur at 120 Hz.
      • Gyroscope data samples occur at 100 Hz.
    • IMU data is packed into a buffer, and covers approximately 165 ms (5 frames at 30 Hz) of time, meaning that each frame contains 33 ms of new data samples, and ~132ms of previous data samples.  This provides redundancy for better tolerance in case of frame error. 
    • Feature tracking data occurs at 30 Hz, and is based on the wide angle lens image.
    • Timestamps for each of the data areas.
    • Camera settings
  2. Wide angle lens back camera image (640x480 resolution, 8-bit greyscale) taken at 30 Hz
  3. Reserved (unused, 1280x96 8-bit data area)
  4. Depth (320x180 resolution, 16-bit data-type) taken at 5 Hz
  5. Perspective lens back camera image (1280x720 resolution, color) taken at 25 Hz
  6. Following these segments of the Y plane (above), the UV plane then occurs, with UV segments appearing in the same order, corresponding to the Y plane.  Because most of the segments do not use color data, the Y values there are constant, and unused.   However, the perspective lens back camera (4 MP color camera) does use the UV plane data, and it starts at line 1392.
The UV plane is visible at the bottom of the "4MP" format image, and the data is ordered as UVUV..    


Data Segment Time Interleaving

New depth data only occurs at 5 Hz, and new color image data occurs at 25 Hz and they do not co-occur.  Since superframes are produced at 30 Hz, this means 5/30 frames contain new depth data, and 25/30 frames contain new color images, with 5 color images preceeding each depth image.   The superframe format does not change however, but some segments are new or stale in each.  

To determine the occurrence of a new depth or big-RGB image, its timestamp should be used.  A timestamp that is not updated since the last frame indicates that image was not updated. 

There is a valid flag bit, but it should only be used to verify that the data contained in the image is valid.  It will not mark new or stale frames.

#define SF2_VTRACK_VALID 1
#define SF2_DEPTH_VALID  2
#define SF2_BIG_VALID    4

  // BitMask indicating which parts bellow are valid.
  uint32_t sf_valid_flags;


Feature Tracking Data


Inside the meta-data area is feature tracking data, calculated in the programmable hardware of the Myriad chip.   It provides motion based feature correspondences across images.  It has the following properties:
  • Per frame, a variable number of features are tracked, up to a maximum of 320.
  • Returning to the same scene will not necessarily result in the same points being tracked, and the IDs will not be the same.  
    • Tracking is flow based, not descriptor-based.
  • Feature tracks are computed independently, without any motion consensus algorithms (such as RANSAC). 
  • Feature correspondences are provided in the "current" and "previous" arrays.
  • "prev_meta" and "prev_xy" arrays together contain the feature "ID" (arbitrarily assigned  ID that is used for the duration of the life of the feature), as well as the x, y position of the feature.  The x, y is referenced as a "raster", so y is the scanline number increasing downwards, and x increases from left to right.  It refers to the features from the Wide Angle Lens area of the previous superframe. 
  • "current_meta" and "current"xy" contain similar data, but for the current frame (the image that is in the Wide Angle Lens area of the same superframe). 
  • At some index "i", i < prev_feature_count,  prev_meta[i] and prev_xy[i] refer to the same feature from the previous frame.
  • At some index "j", j < current_feature_count, current_meta[j] and current_xy[j]refer to the same feature in the current frame.
  • When the ID of the previous and current frame match, then they refer to the same visual feature in both frames, and the difference in position describes the motion.
  • In order to match a feature from the previous frame to the current frame, use the "ID" provided in the prev/current_meta array.
    • Some features will be new, and the feature ID will exist in only the current arrays.
    • Some features will be untracked, and the feature ID only exist only in the previous arrays.
    • Some features will be tracked across frames, and the feature ID will exist in both arrays. 
  • The features are not packed densely into the first elements of the arrays.  
    • Instead the image is logically divided into a grid of 16 cells, arranged as 4x4 isotropic grid.
    • The feature tracking data array can be considered formatted as a set of 16 "sub-arrays":
      • Instead of prev_xy[320], it is more like prev_xy[16][20], where
        • 16 is the # of cells.
        • 20 is the maximum # of features that can occur in any cell.
      • features_per_cell[16] array holds the number of valid features in each cell (cells are indexed in raster order).
      • Then, for each cell, the feature data is packed densely in the "sub" array.
        • If features_per_cell[9] = 2,  then
        • current_xy[9][0] and current_xy[9][1] have valid feature data
        • If the same ID is present in the previous arrays, then it is a match. 


"Small" and "4MP" Superframe Logging Formats


Superframes logged to disk by the Android Player utility are shown below.   Two formats are used to log, "Small" and "4MP".  "4MP" contains all the data of the "Small" format in addition to the image from the back, normal field of view color camera.   
  • On the Left: "Small" Superframe format, displayed with a pixel width of 640. 
  • On the Right: "4MP" Superframe format, displayed with a pixel width of 1280, and 1752 scanlines. 
  • The image file format used is PGM (Specification linked) 
  • Note the PGMs include text comment filled with the "#" character to make the full size of the header 4096 bytes, thus the first byte of Superframe data occurs at byte offset 4096.   When computing byte offsets this difference between on-device superframes and logged superframes should be taken into account (i.e. increment the start pointer for logged data past the comment section). 
  • Byte offsets for each segment were designed to align to 4096 Byte boundaries to facilitate faster disk operations as this matches filesystem blocksizes used on Peanut (resulting in block sized, aligned logging capability for different choices of superframe segments).
As can be seen, the "4MP" format on the right has all the contents of the "Small" format at the top.  The 640x480 wide angle lens image appears left/right doubled at the top because the image width displayed is 1280.   The "4MP" format shown below also shows the UV plane, which appears as the grey area at the bottom of the frame.  Note that only the UV plane portion associated with the perspective frame "large" image camera has useful UV data (since it is the only part of the superframe that has color data, in VUVU ordering).  "4MP" refers to the capability of the large image sensor, although in the project it produces image data of dimension 1280x720.



Parsing Timestamps 

If working with timestamp data directly, the following functions work to parse them.  Note there are multiple superframe versions using different timestamp formats and that the processor tick rate needs to be known for version 0x100 (see below). 
#define PEANUT_TICKS_PER_MICROSECOND 180
enum SuperframeVersion
  : uint32_t {
    kVersion1xx = 0x100
};

// Conversion functions.
// Convert an endian-swapped timestamp into milliseconds.
double ConvertTicksToMilliseconds(
    uint32_t superframe_version,
    const TimeStamp& raw_timestamp) {
  if (superframe_version & SuperframeVersion::kVersion1xx) {
    return (((static_cast<uint64_t>(raw_timestamp.superframe_v2.ticks_lo) << 32)
        | (static_cast<uint64_t>(raw_timestamp.superframe_v2.ticks_hi))) /
        (1000. * PEANUT_TICKS_PER_MICROSECOND));
  } else {
    return static_cast<double>(raw_timestamp.superframe_v1) /
         (static_cast<double>(PEANUT_TICKS_PER_MICROSECOND) * 1000.);
  }
}

double ConvertTicksToSeconds(uint32_t superframe_version,
                             const TimeStamp& raw_timestamp) {
  if (superframe_version & SuperframeVersion::kVersion1xx) {
    return (((static_cast<uint64_t>(raw_timestamp.superframe_v2.ticks_lo) << 32)
        | (static_cast<uint64_t>(raw_timestamp.superframe_v2.ticks_hi))) /
        (1000000. * PEANUT_TICKS_PER_MICROSECOND));
  } else {
    return static_cast<double>(raw_timestamp.superframe_v1) /
         (static_cast<double>(PEANUT_TICKS_PER_MICROSECOND) * 1000. * 1000.);
  }
}


On-Device Superframe Data Structure

A Java example of computing offsets to each of the data areas is shown below.    Note that this sample uses a image "width" of 1280 pixels to describe the offsets in terms of raster line number. 


public class SuperFrame {

  private SuperFrame() {
  } // Prevent instantiation of class.

  // The Superframe's width is 1280 pixels, where each pixel is a byte.
  public final static int SF_WIDTH = 1280;
  // The Superframe's height is 1168 pixels (height can be thought of as
  //   pixels or lines).
  public final static int SF_HEIGHT = 1168;
  // Header data is stored in first 16 lines of a Superframe.
  public final static int SF_LINES_HEADER = 16;
  // A Y plane of wide angle lens (Y contains the luminance of a YUV image)
  // is stored in next 240 lines.
  public final static int SF_LINES_SMALLIMAGE = 240;
  // An Image pyramid is stored in next 96 lines, currently a placeholder.
  public final static int SF_LINES_PYRAMID = 96;
  // A Depth buffer is stored in next 96 lines.
  public final static int SF_LINES_DEPTH = 96;
  // The Y plane of the 4 MP standard field of view camera YUV image
  //   is stored in next 720 lines.
  public final static int SF_LINES_BIGIMAGE = 720;
  public final static int SF_START_LINE_DEPTH = SF_LINES_HEADER
      + SF_LINES_SMALLIMAGE + SF_LINES_PYRAMID;
  public final static int SF_START_INDEX_DEPTH = SF_START_LINE_DEPTH
      * SF_WIDTH;
  public final static int SF_START_LINE_BIGIMAGEY = SF_START_LINE_DEPTH
      + SF_LINES_DEPTH;
  // Size of the Y portion of the 4 MP standard field YUV image.
  public final static int SF_BIG_SIZEY = SF_WIDTH * SF_LINES_BIGIMAGE;
  // Number of bytes in YUV bitmap.
  public final static int SF_BIG_SIZEYUV = SF_BIG_SIZEY + (SF_BIG_SIZEY / 2);

  // The depth buffer is contained in a Superframe as a 320x180 array
  //   of 16 bit (2 contiguous bytes) values.
  public final static int DB_WIDTH = 320;
  public final static int DB_HEIGHT = 180;
  // DB_SIZE is the number of elements, not the number of bytes, in the depth
  //   buffer.
  // An element can be though of as a pixel, where the value is the depth.
  // A single element is a 2 byte int in the Superframe.
  public final static int DB_SIZE = DB_WIDTH * DB_HEIGHT;
}


Detailed Superframe Data Structure (C++)


The following code is used in the camera processing to map data segment locations and is in little endian format when received from the camera buffers, and as ordered in pixels in the logged image data.   A pointer can be assigned to the struct.   The header information also describes the format of the inertial data, feature data, and other meta data in more detail.    It can be useful to read it from the last struct upwards, since it relies on the innermost data structures to be defined first.

// Copyright 2014 Google. Part of the Tango project. CONFIDENTIAL. AUTHORIZED USE ONLY. DO NOT REDISTRIBUTE.
#ifndef COMMON_FRAME_DATA_SUPERFRAME_V2_H_
#define COMMON_FRAME_DATA_SUPERFRAME_V2_H_

#include <stdint.h>

#ifdef NOT_COMPILING_WITH_MDK
// Pack structs on host.
#define PACKED __attribute__ ((__packed__))
#else
#define PACKED
#endif

// Maximum number of features per superframe.
#define MAX_FEATURE_COUNT 320
// Number of 'cells' the pixel pipeline runs fast over.
#define MAX_FEATURE_CELL_COUNT 16
// Number of IMU Packets per superframe.
#define MAX_IMU_COUNT 60

#define MAX_FEATURES_PER_CELL 20

// Super frame data.
// Total sf2_t structure is 20K, this allows both 4K and 1280 16bit scanline
// alignment.
// This block is designed to be 0 % 4096 to allow for fast logging.
// We also align to 1280 16bit scanlines to make MIPI happy.
// Padding allows individual structures to change without disturbing neighbors.
#define SF_WIDTH (1280)
#define SF_HEIGHT (1168)

#define SF_HEADER_LINES 16

#define BIG_RGB_WIDTH 1280
#define BIG_RGB_HEIGHT 720
#define BIG_RGB_SF_LINES 720

#define SMALL_IMG_WIDTH 640
#define SMALL_IMG_HEIGHT 480
#define SMALL_IMG_SF_LINES 240

#define PYRAMID_PAD 8
#define PYRAMID_SIZE  \
    ((SMALL_IMG_WIDTH / 2 + PYRAMID_PAD * 2) * (SMALL_IMG_HEIGHT / 2 + PYRAMID_PAD * 2) + \
     (SMALL_IMG_WIDTH / 4 + PYRAMID_PAD * 2) * (SMALL_IMG_HEIGHT / 4 + PYRAMID_PAD * 2) + \
     (SMALL_IMG_WIDTH / 8 + PYRAMID_PAD * 2) * (SMALL_IMG_HEIGHT / 8 + PYRAMID_PAD * 2))
#define PYRAMID_SF_LINES 96

#define DEPTH_IMG_WIDTH 320
#define DEPTH_IMG_HEIGHT 180
#define DEPTH_IMG_BPP 2
#define DEPTH_IMG_SF_LINES 96

#define SF2_HEADER_SIZE 320
#define SF2_PROFILE_SIZE 320
#define SF2_IMU_SIZE 1920

#define SF2_VTRACK_VALID 1
#define SF2_DEPTH_VALID  2
#define SF2_BIG_VALID    4

// Flags used on small, big and depth frame tearing.
#define SF_TEARING_TRUE 1
#define SF_TEARING_FALSE 0

// Timestamps.
typedef struct PACKED sf2_time_t_ {
    uint32_t ticks_lo;
    uint32_t ticks_hi;
} sf2_time_t_;

union TimeStamp {
    uint64_t superframe_v1;
    sf2_time_t_ superframe_v2;
};

typedef struct PACKED sf_frame_data_t {
  // Could be trigger time or trigger + exposure / 2.
  union TimeStamp timestamp;

  // Image exposure time in uS.
  uint32_t exposureUs;

  // Image color temperature.
  uint32_t color_temp_k;

  // Auto-focus settings.
  uint32_t af_setting;

  // Requested gain.
  uint32_t CAM_GAIN;

  // Requested exposure.
  uint32_t CAM_EXP;

  // Image tear detected flag.
  uint32_t tear_flag;
} sf_frame_data_t;

typedef struct PACKED sf_frame_t {
  // The version of this superframe structure.
  // Lowest 8 bits is minor number.
  // Upper 24 bits are major number.
  // 0x100 - version 1.00
  uint32_t sf_version;
  // The version number of firmware.
  // Upper 16-bits is the year.
  // Lower 16-bits is MMDD#, Month, Day, release # on that day.
  // Version 2014.01211 = 0x07de04bb, 0x07de = 2014, 0x04bb = 1211
  uint16_t firmware_version[2];

  // Superframe frame count from the myriad side.
  uint32_t super_frame_count;

 // BitMask indicating which parts below are valid.
 // For each 16-bit element a bit refers to an image segment
 //   0x01: small image (wide angle camera)
 //   0x02: depth image
 //   0x04: big color image
 // The semantics in each 16-bit element refer to if the image is newly
 // captured in this frame or not, and if the image data in that segment
 // is "valid"  or not.
 //   sf_valid_flags[1]: New flags.
 //   sf_valid_flags[0]: Data OK flags.
 uint16_t sf_valid_flags[2];


  // BitMask indicating which parts bellow are valid.
  uint32_t sf_valid_flags;

  // Frame data from the global-shutter fisheye.
  sf_frame_data_t small;

  // Frame data from the RGB-IR.
  sf_frame_data_t big;

  // Frame data from the depth sensor.
  sf_frame_data_t depth;
} sf_frame_t;

// Feature matching data.
typedef struct PACKED sf_fm_data_t {
  // FM profiling id.
  uint32_t id;

  // FM time in MV clocks.
  uint32_t time;

  // FM number of.
  uint32_t feature_count;
} sf_fm_data_t;  

// Pixel pipeline profile data.
typedef struct PACKED sf_profile_data_t {
  // Processing time for pixel pipeline in ms.
  uint32_t pixel_pipe_time;

  // Processing time for optical flow in ms.
  uint32_t optical_flow_time;

  // Feature maintenance time in ms.
  uint32_t feature_maint_time;

  // Number of features detected by pixel pipeline.
  uint32_t feature_count;

  // FM data for bad feature removal stage.
  sf_fm_data_t fm1;

  // FM data for new feature search stage.
  sf_fm_data_t fm2;

  // FM data for new features admitted stage.
  sf_fm_data_t fm3;
} sf_profile_data_t;

// Raw IMU data.
typedef struct PACKED sf_imu_data_t {
  // IMU timestamp in seconds.
  union TimeStamp timestamp;

  // Enum type decoder.
  enum {IMU_INVALID, IMU_ACCEL, IMU_GYRO, IMU_MAG, IMU_BARO, IMU_TEMP, IMU_LAST} type;

  union {
    // Raw Accel data.
    int16_t accel[4];

    // Raw gyro data.
    int16_t gyro[4];

    // Raw mag data.
    int16_t mag[4];

    // Barometer data.
    uint32_t barometer;

    // Temperature data.
    uint32_t temperature;

    // 32bit array
    uint32_t data32[2];
  } data;
} sf_imu_data_t;

typedef struct PACKED sf_meta_data_t {
  // Rolling identifier of which id
  uint32_t id;

  // Number of frames this feature has been tracked.
  uint32_t age;

  // Original harris score when feature was detected.
  float harris_score;
} sf_meta_data_t;

typedef struct PACKED sf_xy_data_t {
  // X location of the feature.
  float x;

  // Y Location of the feature.
  float y;
} sf_xy_data_t;

typedef struct PACKED sf_vtrack_data_t {
  // MAX_FEATURE_COUNT.
  uint32_t max_features;

  // Number of valid features in prev_meta and prev_xy.
  uint32_t prev_feature_count;

  // Number of valid features in curr_meta and curr_xy.
  uint32_t curr_feature_count;

  // Meta data for the previous feature points.
  sf_meta_data_t prev_meta[MAX_FEATURE_COUNT];

  // Meta data for the current feature points.
  sf_meta_data_t curr_meta[MAX_FEATURE_COUNT];

  // Location of previous features.
  sf_xy_data_t prev_xy[MAX_FEATURE_COUNT];

  // Location of current features.
  sf_xy_data_t curr_xy[MAX_FEATURE_COUNT];

  // Feature matching error.
  float feature_error[MAX_FEATURE_COUNT];

  // Number of features per cell.
  uint32_t features_per_cell[MAX_FEATURE_CELL_COUNT];

  // Fast threshold use for current frame.
  float fast_thresholds[MAX_FEATURE_CELL_COUNT];
} sf_vtrack_data_t;

typedef struct PACKED sf2_header_t {
  // Superframe header.
  sf_frame_t frame;
  uint8_t frame_pad[SF2_HEADER_SIZE - sizeof(sf_frame_t)];

  // Profiling data.
  sf_profile_data_t profile;
  uint8_t profile_pad[SF2_PROFILE_SIZE - sizeof(sf_profile_data_t)];

  // IMU Samples.
  sf_imu_data_t imu[MAX_IMU_COUNT];
  uint8_t imu_pad[SF2_IMU_SIZE - sizeof(sf_imu_data_t) * MAX_IMU_COUNT];

  // vTrack Data.
  sf_vtrack_data_t vtrack;
  uint8_t vtrack_pad[SF_HEADER_LINES * SF_WIDTH -
                SF2_HEADER_SIZE -
                SF2_PROFILE_SIZE -
                SF2_IMU_SIZE -
                sizeof(sf_vtrack_data_t)];
}sf2_header_t;

// Application processor sees Superframes as YUV420SP format.
// The formatting here describes the Y-plane format.
typedef struct PACKED sf2_t {
  // Start Firmware Overlay 0 (Firmware internal YUV400 format).
  // Superframe header.
  sf2_header_t header;
  // Small RGB Data.
  // small_img uses 120 scanlines.
  uint8_t  small_img[SMALL_IMG_WIDTH * SMALL_IMG_HEIGHT];

  // Small image pyramid data.
  // pyramid and pyramid pad use 96 scanlines.
  uint8_t  pyramid[PYRAMID_SIZE];
  // Pad to align pyramid to 20K.
  uint8_t  pyramid_pad[PYRAMID_SF_LINES * SF_WIDTH - PYRAMID_SIZE];

  // Z Depth image 16bits.
  // depth_img and depth_pad use 96 scanlines.
  uint16_t depth_img[DEPTH_IMG_WIDTH * DEPTH_IMG_HEIGHT];
  // Pad to align depth_img to 20K.
  uint8_t  depth_pad[DEPTH_IMG_SF_LINES * SF_WIDTH - DEPTH_IMG_WIDTH *
                     DEPTH_IMG_HEIGHT * DEPTH_IMG_BPP];

  // Big RGB Data Y plane.
  uint8_t big_rgb[BIG_RGB_WIDTH * BIG_RGB_HEIGHT];
} sf2_t;

#endif  // COMMON_FRAME_DATA_SUPERFRAME_V2_H_
